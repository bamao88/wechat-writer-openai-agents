# 产品经理必读：技术选型决定项目生死

在人工智能（AI）飞速发展的时代，产品经理（PM）的工作职责正在发生深刻变革。其中一个关键点就是：**产品经理必须懂技术选型**。这不仅是为了更好地与研发团队协作，更是为了在激烈的市场竞争中，为产品找到最佳的技术路径，甚至可以说，技术选型的决策直接关系到项目的成败。

## 一、为什么产品经理必须介入技术选型？

### 1. 打破研发的“盲目性”与“路径依赖”

在传统软件开发中，产品经理提出需求，研发团队负责方案设计和实现。然而，在 AI 领域，这种模式已经不再适用。

**现象：** 许多研发团队在选择技术方案时，往往只关注 GitHub 上排名前几的开源项目，而忽略了更优的解决方案。例如，有些团队在选择 AI 模型时，仅仅因为某个模型在 GitHub 上 Star 数多，就直接采用，而没有深入比较不同模型的优劣。

**风险：** 这种“路径依赖”容易导致团队陷入“沉没成本”陷阱。一旦选错了技术框架，即使效果不佳，团队也可能因为面子或惯性而不愿推翻重来，最终导致项目走向失败。

**产品经理的作用：** 优秀的产品经理会进行更广泛的调研，例如，张和老师会对比 50 个开源项目，通过分析技术代际差异，找到更先进的架构方案（SOTA）。这种“信息优势”能够帮助团队在预研阶段就纠正方向，避免走弯路。

### 2. 对抗“保守主义”，拥抱新技术

**冲突：** 技术团队通常倾向于使用经过验证的成熟技术，以确保产品的稳定性和可靠性。然而，AI 领域迭代速度极快，过于保守的技术选择往往意味着产品效果不佳，难以满足用户需求。

**产品经理的职责：** 产品经理需要比技术团队更积极地拥抱新技术。例如，当研发团队认为某个功能难以实现时，产品经理可以通过调研竞品（如淘宝 AI 搜索）已经实现的功能，或者查阅最新的论文（Paper），来推动团队突破舒适区，尝试更先进的技术方案。

### 3. 坚持“效果优先于速度和成本”的原则

**原则：** 在预研阶段，产品经理必须坚持“效果（Quality）优先于速度和成本”的原则。

**误区：** 研发团队为了节省成本或追求响应速度（Latency），可能会选择使用小模型（7B）或进行量化剪枝，这会导致产品效果大打折扣，甚至误判产品方向不可行。

**产品经理的决策：** 产品经理必须要求团队先使用最强、最贵的模型（如 GPT-4o, Claude 3.5 Sonnet）来跑通极致效果，建立基线（Baseline）。只有在确认用户价值后，才考虑进行工程优化以降低成本。

## 二、产品经理如何有效介入技术选型？

你不必精通 CUDA 代码，但需要掌握以下四种“非代码”手段来主导技术选型：

### 1. “黑客级”调研：用论文和开源项目说话

- **方法：** 不要只依赖新闻报道，要深入 GitHub 搜索相关开源项目（Repo），并利用 Google Scholar 查阅综述论文（Survey）。
- **案例（Office Copilot）：** 张和老师通过阅读微软的论文，发现 Office Copilot 底层使用了 ODSL (Office Domain Specific Language) 架构。这种对架构的深入理解让他比微软内部非核心员工更懂该产品的原理，从而能够指导自己的团队模仿该架构。
- **案例（OS 模型架构）：** 在规划手机端侧大模型时，研发团队想采用“烟囱式”方案（每个功能训练一个模型）。产品经理通过调研 Google 和 Apple 的最新论文，提出了“基座模型 + Adapter（LoRA 插件）”的架构，比苹果发布同类架构早了 6 个月。

### 2. 评测驱动（Evaluation Driven）：用数据否决烂方案

- **逻辑：** 选型的好坏不是研发团队说了算，而是评测集说了算。
- **操作：** 建立包含极端情况（Corner Case）的高质量评测集。如果研发团队选择的模型无法通过评测集，或者表现不如竞品（如 Photo AI），产品经理就有足够的底气要求更换模型。
- **案例（AI 写真工程事故）：** 算法团队训练的模型效果很好，但工程团队部署后效果变差。产品经理通过“控制变量法”排查，发现是工程团队在部署 Stable Diffusion WebUI 时，采样器（Sampler）等参数设置错误。不懂参数选型的产品经理会被忽悠，而懂行的产品经理则能够及时发现问题并解决问题。

### 3. 亲手复现（Hands-on）：用 AI Coding 验证可行性

- **新趋势：** 现在的 AI Coding 工具（如 Cursor, Cloud Code, v0）非常强大，产品经理完全可以自己编写代码来复现一个 Demo。
- **价值：** 当你使用 AI 工具花几个小时就把一个核心流程跑通，并看到了日志（Log）和中间产物时，你就拥有了与研发团队平等对话的权利。你可以明确告诉研发团队：“我看过 Log 了，这个 Prompt 在这一步有问题”，而不是凭空猜测。
- **策略：** 对于拿不准的技术方案（如 Workflow vs. Agent），产品经理可以自己使用 Coze 或 Dify 搭建一个，或者让 AI 编写代码运行一个，直接对比效果。

### 4. 架构图梳理（Pipeline Analysis）：发现逻辑漏洞

- **逻辑：** 即使不写代码，产品经理也必须能够画出核心算法的数据流向图。
- **案例（运动追焦）：** 产品经理通过梳理架构图发现，算法处理的是全画幅传感器数据，而用户屏幕上看到的是裁切后的 16:9 画面。这种“输入不一致”的架构错误，只有懂流程的产品经理才能发现。

## 三、常见的技术选型决策点

### 1. 模型选型

- **预研期：** 必须选择 SOTA（最强模型），不要为了省钱而妥协。
- **落地期：** 根据评测结果，尝试使用小模型微调或蒸馏来替代大模型。

### 2. RAG vs. Long Context

- **判断：** 产品经理需要判断业务主要依靠检索（RAG）还是长文本理解。如果是 RAG，要关注检索（Retrieval）和生成（Generation）的分段评测。

### 3. Agent vs. Workflow

- **判断：** 是让 AI 自主规划（Agent），还是将每一步的逻辑写死（Workflow）？
- **决策：** 如果业务流程固定，优先使用 Workflow（更稳定）；如果场景极度发散，再考虑 Agent。不要为了使用 Agent 而强行使用。

## 总结

产品经理参与技术选型，本质上是掌握“定义的权力”。你不必知道 Transformer 的数学公式，但你必须知道：

- 现在的技术天花板在哪里（SOTA）？
- 业界的最佳实践架构是什么（Best Practice）？
- 以及如何通过评测去验证它（Evaluation）。

**心态：** 宁愿在技术上“过度自信”（乐观拥抱新技术），也不要被研发的“过度保守”牵着鼻子走。因为在 AI 时代，选择（Selection）大于努力（Effort）。

在 AI 时代，产品经理只有深入理解技术选型，才能在激烈的市场竞争中，为产品找到最佳的技术路径，最终实现产品的成功。
